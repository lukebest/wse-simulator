TDM Flattened Butterfly：Color 数 × Mesh 规模 × FB 规模

基于 wsesim 物理 max-load 扫描（= 贪心着色下界 C）。约束：kⁿ = S²（S×S mesh，N=S² PE）。↓ Color 子拓扑生成过程

Color = 24 时，最大支持多大 Mesh？

32 × 32（1024 PE）

在全部合法 (k,n) 配置中，最大且 C≤24 的为 (k,n)=(2,10)：C=21，逻辑链路 10240 条，逻辑直径 10 跳。

注意：C=24 是上界；具体 (k,n) 需 C≤24 且满足 kⁿ=S²。不存在单一「mesh 规模→C」映射——同一 S 上不同 (k,n) 的 C 可差 4× 以上。

核心规律

C = max_{物理有向边 e} |{ 逻辑链路 ℓ : XY_route(ℓ) ∋ e }| = color_lower_bound
  • C 不等于逻辑维度 n、逻辑链路数 L、或 mesh 边总数。
  • C 等于最拥塞物理边上的逻辑路径条数（TDM 周期下界；已测配置贪心均达到）。
  • 约束：N = S² = kⁿ，L = N·n·(k−1)。
  • 同 N 下：k↑ n↓ → L↑ 且 C↑（长组内链路多）；k=2 n↑ → C 缓增，可撑更大 mesh。

Color 子拓扑：生成过程（源码路径）

「Color 子拓扑」不是在物理 mesh 上再建一张独立图，而是：枚举 FB 逻辑长链路 → 映射 XY 物理路径 → 全局贪心着色 → 每个 color id 对应一组物理边互不冲突的逻辑链路。实现见 wsesim/network/tdm_coloring.py、topology/tdm_flat_butterfly.py。

定义

在 cycle t，仅 color = t mod C 的逻辑链路可在 router 出口前进（Color 门控在 router，不在 link）。 Color c 的子拓扑 = 所有 color_of_logical[(u,v)] = c 的有向逻辑弧；约束为同 color 内任意两条逻辑链路 不得共享同一条物理有向边 (p_src, p_dst)。

link_active[(p_src,p_dst)][c] → 该物理边在 color c 时隙由哪条逻辑链路占用（或 None）

四层流水线

Layer 0 · 物理 Mesh2D — 4 邻接图，节点 id 行优先；仿真中 link 带宽 / latency / VC 均在此层。
Layer 1 · 逻辑 FB 长链路 — logical_links()：每维上节点与同维其余 k−1 个坐标不同的节点建立有向逻辑弧 (u,v,dim)。
Layer 2 · XY 物理路径 — physical_path(u,v)：每条逻辑弧映射为 mesh 上有向边序列（先列后行 DOR）。
Layer 3 · 着色 + 子拓扑 — assign_colors() → ColorPlan；_build_link_active() 导出各 color 的激活表。
Layer 4 · 运行时 TDM — TDMClock(period=C) + router per-color buffer；flit 等待 current_color(t) == flit.color。

Step 1 — 枚举逻辑 FB 长链路

入口：TDMFlatButterfly.coloring() 调用 logical_links()。对 kn 配置，节点 id 与超立方坐标互转（to_coords / to_node）。 

for node in range(N):
    for dim, peers in logical_neighbors(node).items():
        for peer in peers:
            links.append((node, peer, dim))   # 有向逻辑长链路 + 所属维
配置逻辑链路 L含义
4×4 (4,2)9616×2×3，每维 3 个逻辑邻居
4×4 (2,4)6416×4×1，每维 1 个逻辑邻居（位翻转）
8×8 (2,6)38464×6×1

矩形 mesh（非 kn）：见 RectFlatButterfly / scripts/generate_tdm_fb_rect_docs.py — dim0 = 同行全连（radix=cols），dim1 = 同列全连（radix=rows）；逻辑链路数 = N(cols−1) + N(rows−1)。

Step 2 — 逻辑链路 → XY 物理路径

对每条出现在 logical_links 中的 (u,v)，计算 mesh 最短路式的 XY 有向边序列（非 BFS，固定先水平后垂直）： 

physical_path(src, dst):
    先沿列从 sc → dc（每步一条有向 mesh 边）
    再沿行从 sr → dr

同时收集 physical_links()：mesh 上全部双向有向边（4×4 → 48 条）。着色只关心逻辑链路的路径是否经过这些边。

Step 3 — 物理边负载与 Color 下界

着色前扫描：统计每条物理有向边被多少条逻辑链路的 XY 路径穿过。

for logical_link in logical_links:
    for edge in physical_paths[logical_link]:
        load_per_link[edge] += 1

color_lower_bound = max(load_per_link.values())   # 至少 C ≥ 此值

直觉：同一时刻一条物理边只能承载一条逻辑长链路的数据；若某边被 L 条逻辑路径共享，TDM 至少需要 L 个 color 轮转。 这与上文「核心规律」公式一致；下文贪心初始 color_count = lower_bound。

Step 4 — 贪心着色 assign_colors

  1. 排序：物理路径更长的逻辑链路优先分配（longest-path-first，提高装箱质量）。
  2. 尝试：对链路 ℓ，从小到大试 color c；当且仅当 ℓ 的 XY 路径上每条物理边在 color c 上仍为 free，才将 ℓ 放入 c。
  3. 扩容：若现有 color 均放不下，则 C += 1，为新链路独占新 color。
  4. 输出：color_of_logical[(u,v)]、logical_dims、load_per_physical_link。
for logical_link, dim in ordered_by_path_length_desc:
    for color in range(color_count):
        if all(path_edges_free_at[color] for edge in path):
            assign logical_link → color; mark edges busy; break
    else:
        color_count += 1; assign to new color

已文档化 square-mesh 配置中，贪心结果均达到下界（无额外扩容）：4×4 (2,4) C=2，(4,2) C=4；8×8 (2,6) C=5 等。

Step 5 — 构建 link_active（各 Color 子拓扑的数据结构）

着色完成后，对每个 color c 汇总其全部逻辑弧，并登记每条物理边在 c 下的占用者；若同一边上两条不同逻辑链路争用同一 c，则抛 ValueError（验证器见 visualize_tdm_coloring_4x4.py）。 

for logical_link, color in color_of_logical.items():
    for edge in physical_paths[logical_link]:
        link_active[edge][color] = logical_link   # 该 color 子拓扑在 edge 上的激活链路

可视化 HTML（如 k2_n4）即：对每个 c，在物理 mesh 上绘制 color c 的全部逻辑弧； overview 图边标签 load(e) 是争用次数，不是 color 编号。

运行时：子拓扑如何「开门」

  • UnifiedNetwork 读取 topology.coloring()，设 Router.color_count = C，创建 TDMClock(period=C, slot_cycles=…)。
  • TDMFlatButterflyRouting 为每个逻辑 hop 查 color_of_logical[(u,v)]，flit 携带 color tag。
  • Router pipeline 入口：若 tdm_clock.current_color(t) ≠ flit.color，自旋等待（累加 color_buffer_wait_cycles）。
  • 含义：cycle t 只有 color t mod C 对应的那张子图在 router 侧可转发；其余 color 的逻辑链路排队。

进阶：多跳路由与路径单色约束

默认贪心按单条逻辑链路着色。维序路由（如 k2_n4 的 0→6 = 0→2→6）含两条逻辑 hop，可能被分到不同 color → 单 flit 无法保持单色。 此时需 assign_colors_for_routes：对 force_monochrome_paths 强制同一路径所有 hop 共用一个 color，放不下则增大 C 重试 （见 visualize_tdm_coloring_4x4.py 0→6 示例）。 AllGather ND 在 k=2 组内通信为单逻辑 hop，不受此影响；多 hop 端到端路由才需 path-aware 着色。

一句话

Color 子拓扑生成 = 在物理 mesh 上列出全部 FB 逻辑长链路 → 为每条算 XY 物理路径 → 用 max-load 定 C 下界 → 贪心装箱（长路径优先，同 color 不共享物理边）→ 得到 color_of_logical 与 link_active； 每个 color id 即一张可同时激活、物理无冲突的逻辑通信子图。

经验公式

C = ⌊k²/4⌋
2-flat (n=2)，S×S mesh，k=S（实测 k=4…32）
C = (k−1)²
例：8×8 (4,3) → C=9
C ≤ 24 → k≤9
n=2 时：9×9 可行，10×10 超限 (C=25)
k=2 族
n=4→C=2, n=6→5, n=8→10, n=10→21

跨配置 Color 规律

  • k-ary, n=2（2-flat）：Color 类由 k 节点组内逻辑距离决定——C0=邻接，奇数距离拆为端点/中间两 Color，最大距离单独一 Color；C ≈ k（8×8 为 16）。
  • k=2（binary hypercube）：基础 2-Color 由坐标参考位奇偶决定；mesh 变大时跨维路径冲突增加，C 从 2 扩展至 5（8×8）或更高。
  • 4 节点组规则：4×4 (4,2) 与 8×8 (4,3) 共享同一距离—Color 映射；后者因 3 维叠加 + 更大 mesh，C 从 4 升至 9。
  • 单 Color 内：同一时隙所有链路物理边互不冲突；跨 Color 合并 = 完整 FB 逻辑拓扑。

设计含义（Cerebras 24 Color）

  • 若坚持 2-flat 高 radix（逻辑 2 跳、组内全连）：mesh 不宜超过 9×9。
  • 若可接受 binary 多 flat（逻辑 10 跳）：单程序可覆盖 32×32 = 1024 PE（C=21，余 3 个 color 余量）。
  • 8×8 (8,2) 用 C=16，余 8 个 color —— 与现有 4×4 (4,2) 用 C=4 占满下界 的比例类似。
  • 选 (k,n) 是在 逻辑直径、链路数 L、TDM 周期 C 之间的权衡，不能只看 mesh 边长。
