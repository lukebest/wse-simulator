矩形 Mesh AllGather：Mesh2D vs 矩形 2-flat FB vs 限制版 Hypercube FB

Mesh 尺寸：6×8、12×16、14×14。主对比算法：nd_dimension_exchange_allgather (slot=1)。

指标说明（读表前必读）

为什么 color_wait 可以远大于 makespan？
两者量纲不同，不能直接比较大小。makespan 是仿真时钟的墙钟时间（最后一个包完成时的 env.now，即关键路径长度）。 color_wait 是全网络累加计数：每个 router 上、每个 flit 在 per-color buffer 出口等待「当前 TDM 时隙 == 该 flit 的 Color」时，每 spin 1 cycle 就 +1，最后把所有 router 的等待求和。 大量包/ flit 在并行等待，累加值自然远大于墙钟。例：12×16 满 FB ND 1KB — makespan=9225，color_wait=1,104,889（≈120×），表示并行等待总量大，但不代表单包等了 120 万 cycle。
TDM/Mesh ≈ 0.03×、0.07× 的时延收益从哪来？
比值只看 makespan（端到端完成时间），与 color_wait 累加值无关。限制版 hypercube FB 的优势主要来自：
  1. Color 数 C 更少（5/10 vs 16/64/49）→ 每跳等正确 TDM 时隙的周期更短，关键路径上 color 相关等待更少。
  2. ND 流量模式不同：限制版按 (k=2, n) 超立方分组，每组仅 2 节点交换；Mesh/满 FB 按混合基 k_dims（如 16×12）分组，组内全对全通信，包数/flit 数多一个数量级（如 12×16 Mesh 342k flits vs 限制版 5k flits）。
  3. 平均包延迟更低：限制版 avg_lat 约 39–118 cycles，Mesh 350–5585 cycles（见完整结果表）。
  4. 逻辑跳数更少：稀疏 hypercube 链路 + 维度序路由，单包物理跳数通常短于 Mesh 分组交换在大组内的多跳转发。
注意：限制版用更少的逻辑链路换更少的 Color；满 FB 在 14×14 上 C=49 反而比 Mesh 慢（makespan 8×），说明Color 数过多的代价可以超过 FB 拓扑的带宽优势。
msg（msg_bytes）
实验配置的标称消息大小（字节），用于标识测试点，写入 CSV 的 msg_bytes 列。
simulated_msg（simulated_msg_bytes）
仿真中实际注入的 payload 大小。通常与 msg 相同。 若标注 (scaled)，表示 direct_allgather 在 256KB 点时按 payload / N 缩放（6×8 除以 48，12×16/14×14 除以 64），以降低 O(N²) 全对全流量的仿真成本；此时 msg=262144 但 simulated_msg 仅为几千字节。
color_wait（color_buffer_wait_cycles）
Router 侧 TDM 等待：flit 从 per-color ingress buffer 取出后、进入 switch pipeline 之前，等待全局 TDM 时钟转到该 flit 的 Color 时隙。仅 TDM 拓扑非零；Mesh2D 恒为 0。全 router 累加。
link_wait（link_wait_cycles）
物理链路带宽争用：flit 在 Link 的 resource.request() 上排队等待（与 Mesh 相同机制）。 TDM Color 时隙不在 link 侧重检——router pipeline 已按 Color 放行。无其他 flit 争用同一链路时为 0。
makespan（makespan_cycles）
AllGather 流量全部完成时的仿真时间（cycles）。端到端延迟对比以这一列为准；TDM/Mesh 比值 = TDM makespan ÷ Mesh makespan。
avg_lat
所有完成包的平均单包延迟（cycles），反映典型包的体验，与 makespan（最慢包决定）不同。
