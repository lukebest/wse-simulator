1) 关键结论

  • 最佳组合是 `tdm_fb_k2_n4 + nd_dimension_exchange_allgather + slot=1`，三档消息大小都优于 Mesh2D（约 1.77×~1.84×）。
  • `tdm_fb_k4_n2` 在两种算法下均慢于 Mesh2D，slot=1 时约慢 1.32×~1.50×；slot=4 会进一步恶化到约 1.92×~2.18×（以 nd 算法对比 mesh nd）。
  • 算法选择影响很大：在同一拓扑（slot=1）下，`nd_dimension_exchange_allgather` 相对 `direct_allgather`：
    • Mesh2D：约 3.61×~3.75× 更快
    • k4_n2：约 3.60×~3.68× 更快
    • k2_n4：约 7.82×~8.44× 更快
  • slot 从 1 增大到 4 的惩罚呈现“拓扑相关性”：
    • k4_n2：makespan 增加约 1.45×~1.65×（非常敏感）
    • k2_n4：增加约 1.00×~1.21×（敏感性低，尤其大消息几乎不变）

2) 为什么会这样（结合统计项解释）

  • k4_n2 的瓶颈主要是时分等待：`color_buffer_wait_cycles` 和 `link_wait_cycles` 在大消息下很高。例：256KB + direct + slot=4，`color_buffer_wait_cycles=751190`，`link_wait_cycles=310675`。
  • k2_n4 的色等待明显更低：256KB + nd + slot=1 时，`color_buffer_wait_cycles=14792`（相比 k4_n2 的 71550 显著更小），使端到端完成时间更短。
  • slot 变大直接拉低链路利用率：k4_n2 的 `avg_link_util` 从约 0.075/0.083 降到约 0.050/0.057；k2_n4 也下降，但性能退化更温和。
  • Mesh2D 中 `link_wait_cycles` 与 `color_buffer_wait_cycles` 为 0（无 TDM 颜色门控），其时延主要由路由路径长度和 pipeline 周期累积决定。

4) 工程建议

  • 如果目标是 AllGather 完成时间，优先采用 k2_n4 + nd_dimension_exchange_allgather + slot=1。
  • 若必须使用 k4_n2，建议优先降低 slot 大小，并进一步做路由/调度 co-design 来减少跨 color 等待。
  • 后续可以加入 `slot_cycles={2,8}`、更多消息粒度、以及统一“等总字节”负载对比，验证结论在更广参数下的稳定性。
