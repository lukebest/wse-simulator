AllGather 算法详解：direct_allgather vs nd_dimension_exchange_allgather

1) 总览：两种算法的本质区别

二者都是 AllGather（每个节点最终持有全部专家的 payload 片段），但流量调度策略完全不同：

direct direct_allgather

单阶段、全局全对全。每个源节点在同一时刻向所有其他节点发送自己的 chunk。无维度分组、无 stage 串行。

包数 ≈ N × (N−1) × num_experts

复杂度 O(N²)，适合小规模（源码默认 shards≤8 时自动选用）。

ND nd_dimension_exchange_allgather

多 stage、组内全对全。按拓扑维度串行：每个 stage 仅在同维同组的 k 个节点内做 direct allgather，组间通过多 stage 间接传播。

包数 ≈ Σdim Gdim × kdim(kdim−1) × num_experts

流量通常远小于 direct；TDM FB 拓扑的默认 AllGather 算法。

关键洞察：算法决定谁向谁、何时、发多少；NoC 类型决定这些包如何在物理网络上路由、是否受 Color 门控。 同一算法在不同 NoC 上表现可以天差地别——尤其是 direct_allgather 在 TDM 上会因 O(N²) 流量与 Color 等待叠加而急剧恶化。

2) direct_allgather 算法

源码逻辑

def _direct_allgather(nodes, payload_bytes, num_experts):
    s = len(nodes)
    chunk = ceil(payload_bytes / s)          # 每节点持有 1/s 的 payload
    for expert_id in range(num_experts):
        base = expert_id * s                 # 同一 expert 内所有包同 delay
        for src in nodes:
            for dst in nodes:
                if src != dst:
                    traffic.append(packet(src, dst, chunk, "allgather", base))

行为特征

  • Chunk 划分：chunk = ⌈payload / N⌉。每个节点只广播自己负责的那一份，接收方在仿真中通过多包汇聚得到完整 tensor（仿真层不建模本地 concat，只建模网络流量）。
  • 时间模型：同一 expert 内所有 N(N−1) 个包共享同一 delay_cycles = expert_id × N，即完全并发注入，无 stage 屏障。
  • 正确性直觉：经过一轮，每个节点从其余 N−1 个节点各收到 1 份 chunk，本地已有自己的 chunk → 凑齐 N 份，AllGather 完成。
  • 流量规模：随 N² 增长。64 节点、1 expert 即 4032 个包；256KB 消息在 8×8 仿真中产生 ~24 万 flit 量级（见 benchmark 报告）。

与 Mesh / TDM 文档的映射

项目文档规定：Mesh2D 基线的 AllGather 默认算法为 direct_allgather； TDM FB 基线为 nd_dimension_exchange_allgather（见 docs/tdm_flat_butterfly_4x4.md）。

3) nd_dimension_exchange_allgather 算法

源码逻辑

def _nd_dimension_exchange_allgather(nodes, payload_bytes, num_experts, topology_hint):
    groups_by_dim = _resolve_groups_by_dimension(nodes, topology_hint)
    n = len(groups_by_dim)                   # 维度数
    chunk = ceil(payload_bytes / len(nodes))
    for expert_id in range(num_experts):
        offset = expert_id * n
        for dim in range(n):                 # 按维度串行 stage
            for group in groups_by_dim[dim]:
                for src in group:
                    for dst in group:
                        if src != dst:
                            traffic.append(packet(src, dst, chunk, "allgather", offset))
            offset += 1                      # 下一维 stage delay +1

分组解析（topology_hint 驱动）

拓扑类型hint 字段分组方式示例（16 节点 4×4）
Mesh2D / 矩形 Mesh k_dims=[cols, rows], cols 混合基坐标：dim0=列方向同行组，dim1=行方向同列组 k_dims=[4,4] → 每维 4 组 × 每组 4 节点
TDM FB（kn） k, n 或 hypercube_coords 超立方坐标：固定某一维、其余维相同者为同组 k2_n4 → 4 维，每维 8 组 × 每组 2 节点
限制版 Hypercube FB k=2, n 同上，但仅保留子矩形内完整 XY 路径的逻辑链路 12×16 限制版 → 组大小仍为 2，组数更少

Stage 语义

第 dim 个 stage 完成后，在该维度的每个 k-节点组内，所有成员已持有组内全部 chunk。 经过全部 n 个 stage，信息沿各维扩散，最终每个节点持有全局 N 份 chunk（与 direct 目标相同，但路径分 stage、组内全对全）。

与 AllReduce 版 ND 的区别：nd_dimension_exchange_allreduce 在组内用 recursive halving-doubling 或 ring； AllGather 版则在每个 stage 的每个组内直接做 mini direct allgather（组内 k(k−1) 条边），不做 reduce。

Delay / 并发模型

  • 同一 stage（同一 offset）内：所有组的组内包并发注入。
  • 不同 stage：通过 offset += 1 错开 1 cycle（极简 barrier 模型），形成 n 个逻辑阶段。
  • 不同 expert：offset 基址为 expert_id × n，专家间也按 n 个 cycle 错开。

5) 分组交换 NoC（Mesh2D 物理 mesh）

Mesh2D 「分组交换 NoC」指物理层为 2D mesh 邻接图、路由为 DimensionOrderRouting（BFS 最短路）、无 Color / 无 TDM 时隙 的网络。 算法层面的「分组」体现在 ND 的 stage 划分；网络层不感知逻辑分组，只按 src/dst 做 packet 路由。

物理与路由

  • 拓扑：4 邻接 mesh，节点 id 行优先编号（见 4×4 文档）。
  • 路由：DimensionOrderRouting → BFS 在 mesh 图上求下一跳，等价于曼哈顿最短路径（非严格 XY，但 hop 数最优）。
  • 链路：普通 Link，无 color 字段；router color_count=0，color_buffer_wait_cycles=0。
  • 争用：仅来自 VC/ buffer / link 带宽，无 TDM 旋转等待。
direct_allgather @ Mesh： N(N−1) 个包同时注入，全部走 mesh 最短路。远距离节点对（如 0→15）需 ~6 hop，瞬时链路负载高。 优点：无 color 等待；缺点：O(N²) 包数导致 buffer/link 争用随 N 急剧上升。
nd_dimension_exchange @ Mesh： 使用 k_dims=[cols, rows]（矩形 mesh）分组。以 4×4 为例：
  • Stage 0（列维）：4 条「行」组，每组 4 节点，组内 4×3=12 包/组 → 48 包
  • Stage 1（行维）：4 条「列」组 → 再 48 包
  • 合计 96 包 vs direct 的 240 包（2.5× 更少）
组内通信多为同行/同列相邻或短距，物理 hop 更短、争用更局部；stage 串行使峰值并发低于 direct。 仿真中 4×4 mesh ND 的 makespan 约为 direct 的 1/3~1/6（见 4×4 报告）。

6) Color 时分 NoC（TDM Flat Butterfly overlay）

TDM FB 物理层仍是同一 mesh，但在其上叠加 kn 逻辑 FB：每维上节点与同维其余 k−1 个节点有逻辑长链路， 映射为 XY 物理路径；全局边着色得到 ColorPlan（C 个 color，周期 TDM 轮转）。

Color 调度机制

  • 着色：每条逻辑链路 (u→v) 分配固定 color；同一 color 内逻辑链路不共享物理有向边（见 assign_colors）。
  • 时钟：TDMClock，cycle t 上仅 color = t mod C 的逻辑链路可前进。
  • Router 门控：flit 进入 per-color ingress buffer 后，在 pipeline 入口自旋等待直到 current_color == flit.color（累加 color_buffer_wait_cycles）。
  • 路由：TDMFlatButterflyRouting 按逻辑维序逐段跳转：每段绑定一条逻辑链路的 color 与 XY 物理 hop 序列。
  • Link 层：TDMLink 不在链路上二次检查 color（门控仅在 router）。
direct_allgather @ TDM FB： 强烈不推荐。 仍产生 N(N−1) 个包，但每个包可能跨多条逻辑链路、多个 color。 大量包错过当前 color 时隙 → router 侧 color 等待爆炸（4×4 k4_n2 上 color_wait 可达 105 量级，makespan 为 Mesh direct 的数倍）。 O(N²) 流量与 C 周期 TDM 叠加，64 节点下 TDM 全面慢于 Mesh（8×8 报告结论）。
nd_dimension_exchange @ TDM FB： 推荐组合。 分组与逻辑 FB 维度对齐（k, n 来自 topology_hint）。 每个 stage 只激活某一逻辑维的组内交换，且组内 k 较小时（如 k=2 仅 1 对包/组）流量极低。 例：k2_n4 在 4×4 上 n=4 stage，每 stage 8 组 × 2×1 = 16 包，四 stage 共 64 包（vs direct 240 包）。 逻辑链路 color 与 stage 天然部分对齐，color 等待可控；k2_n4 + slot=1 在 4×4 上 makespan 约为 Mesh ND 的 0.55×。

TDM 上 ND 分组（k2_n4, 16 节点）

节点 id 与超立方坐标：node = Σ coord[i]·2i。每维 8 个二元组 {0,1}、{2,3}、…、{14,15}。 Stage 0~3 各做 8 组 × 2 节点组内 direct allgather。

Color 数 C 的影响

(k,n) 示例节点典型 C对 ND AllGather 的影响
k2_n416较小4 stage × 小组交换，color 等待低，最优 TDM 组合之一
k4_n216≈42 stage 但组内 k=4 全对全，单 stage 包更多

数据流对比（概念）

direct_allgather:
  [所有节点] ──同时──► 全对全 unicast ──► Mesh/TDM 物理路径

nd_dimension_exchange_allgather:
  for dim in 0..n-1:
      for each k-node group in dim:
          组内 direct allgather (k×(k-1) 包)
      ──► 每包走 Mesh 最短路 或 TDM 逻辑维序路由 + Color 门控

包数推导（k2_n4, ND）

Gd = N / k = 16/2 = 8 组/维  |  每维包数 = Gd × k(k−1) = 8 × 2 = 16  |  n = 4 维合计 = 4 × 16 = 64 包

k4_n2 同为 2 维：2 × (4 组 × 4×3) = 2 × 48 = 96 包，与 mesh ND 包数相同但分组几何不同（超立方 vs 行列混合基）。

9) 选型建议

  1. Mesh2D 小集群（N≤16）：可用 direct_allgather 作简单基线；追求更低 makespan 用 nd_dimension_exchange_allgather + k_dims=[cols,rows]。
  2. Mesh2D 大集群（N≥64）：避免 direct（O(N²) 仿真不可承受）；仅测 ND 分组交换。
  3. TDM Flat Butterfly：始终用 nd_dimension_exchange_allgather；优选小 k、多 n（如 k2_n6 on 8×8）。
  4. 评估指标：端到端对比看 makespan_cycles；color_buffer_wait_cycles 是全网累加，仅用于分析 TDM 调度压力，不能与 makespan 比绝对值大小。
  5. 算法与 NoC 配对：Mesh 文档配 direct/2d_ring；TDM 文档配 nd_dimension_exchange——这是有意为之的公平对比基线，跨 NoC 比较算法时需固定拓扑与 hint。
